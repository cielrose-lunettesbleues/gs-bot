# Session Dev — 2026-06-10 : Migration SaaS multi-tenant

## Objectif de départ

Transformer le projet depuis son état initial (bot standalone mono-utilisateur piloté par `.env`) vers une **application SaaS multi-tenant** où plusieurs streamers peuvent se connecter à la même instance via OAuth Twitch, sans jamais toucher un fichier de configuration.

---

## Architecture résultante

```
Streamer → /setup (opérateur) → OAuth Twitch → /dashboard → OBS Browser Source
```

Point d'entrée principal : `src/server.ts` (`npm run dev`)  
L'ancien mode standalone `src/app.ts` est exclu de la compilation (`tsconfig.json`).

**Stack** : Hono + SQLite (better-sqlite3) + tmi.js + SSE overlay

---

## Fichiers créés

### `src/server.ts`
Serveur principal Hono multi-tenant. Remplace complètement `app.ts` pour le mode SaaS.

Routes :
- `GET /setup` — wizard premier lancement (redirige vers `/` si déjà configuré)
- `POST /setup` — sauvegarde les credentials Twitch, met à jour `oauthConfig` en mémoire
- `GET /` — login page ou redirect vers `/dashboard` si authentifié
- `GET /dashboard` — dashboard par utilisateur (session requise)
- `GET /auth/twitch` — initie le flow OAuth Twitch (state CSRF cookie)
- `GET /auth/twitch/callback` — échange le code, crée session, démarre le bot IRC en arrière-plan
- `POST /auth/logout` — invalide la session
- `GET /overlay/:channel` — page HTML OBS Browser Source (sans auth)
- `GET /overlay/:channel/events` — SSE stream pour l'overlay OBS
- `GET /api/status` — état du tenant (config, queue, approbations, overlay, Twitch)
- `GET /api/history` — historique de lecture
- `PATCH /api/config` — modification de config à chaud
- `POST /api/queue/stop` — stop d'urgence
- `POST /api/cooldown/reset` — reset cooldown
- `POST /api/approve/:username` — approbation mod
- `POST /api/deny/:username` — refus mod
- `POST /api/simulate` — simulateur de chat depuis le dashboard

### `src/config/serverConfig.ts`
- `loadServerConfig()` — fusionne `process.env` + `data/server-config.json` (le fichier surcharge l'env)
- `loadPersistedEnv()` — lit `data/server-config.json` si existant
- `saveServerOAuthConfig()` — écrit le JSON persisté après le wizard
- Toutes les variables Twitch ont `.default("")` → le serveur démarre sans crash même sans `.env`

### `src/views/setupHtml.ts`
Wizard de premier lancement (dark theme). Génère automatiquement la redirect URI depuis `window.location.origin`. Instructions pas-à-pas pour créer l'app Twitch. Sauvegarde via `POST /setup` et redirige vers `/`.

### `src/views/dashboardHtml.ts`
Dashboard multi-tenant complet :
- Header avec channel name, status dot, badge file d'attente
- Panel "OBS Browser Source" avec URL overlay pré-rendue + bouton Copier
- Configuration toggles (sub-only, mod-only, cooldown, approbation)
- Actions (stop urgence, reset cooldown)
- Liste d'approbations en attente
- Historique récent (tableau)
- Simulateur de chat intégré
- Refresh automatique toutes les 5s

### `src/views/loginHtml.ts`
Page de connexion avec bouton "Connexion avec Twitch" (Twitch purple). Affiche les instructions `.env` si OAuth non configuré.

### `src/tenant/tenantManager.ts`
`TenantManager` : crée et met en cache les services isolés par `userId` :
- `PlaybackQueue`, `CooldownService`, `ApprovalService`, `BlacklistService`, `HistoryService`
- `OverlayBroadcaster`, `TwitchBotManager`, `CommandRouter`
- Config runtime (`TenantRuntimeConfig`) partagée par référence entre tous les services
- `persistConfig()` : persiste en DB et met à jour la référence en mémoire sans restart

### `src/auth/oauthHandler.ts`
- `buildAuthUrl()` — construit l'URL d'autorisation Twitch avec scopes `chat:read chat:edit channel:read:redemptions user:read:email`
- `exchangeCode()` — échange le code OAuth contre les tokens
- `refreshAccessToken()` — refresh automatique
- `fetchUserInfo()` — appelle `/helix/users` pour récupérer login et display name
- `getUserFromSession()` — lit la session DB, refresh le token si < 5min restantes

### `src/auth/sessionMiddleware.ts`
- `sessionMiddleware()` — middleware Hono qui lit le cookie `gs_session` et popule `c.get("sessionUser")`
- `requireAuth()` — retourne `SessionUser | null`
- Helpers cookie sans dépendance externe : `getCookie`, `setSessionCookie`, `clearSessionCookie`

### `src/db/database.ts` (étendu)
Schéma SQLite complet :
- `users` — profils Twitch + tokens OAuth
- `sessions` — sessions avec expiration (30 jours par défaut)
- `tenant_configs` — configuration par utilisateur avec INSERT OR IGNORE (valeurs par défaut)
- `history` — historique de lecture par utilisateur
- `blacklist` — liste noire par utilisateur

## Décisions d'architecture

### `oauthConfig` comme objet mutable
Le middleware `sessionMiddleware(db, oauthConfig)` capture `oauthConfig` par référence. Après le wizard `/setup`, `oauthConfig.clientId` etc. sont mutés en place → pas besoin de restart serveur.

### Bot IRC non-bloquant
`tenant.twitchBotManager.start()` est lancé en fire-and-forget après la création de session dans le callback OAuth. Le streamer atteint toujours le dashboard même si tmi.js est lent ou échoue à connecter.

### URL overlay : double rendu
Le span `#overlay-url` est pré-rempli côté serveur avec `/overlay/${channelName}`. Le JS le remplace avec l'URL complète (`window.location.origin + path`). Robuste même si JS échoue partiellement.

### Services isolés par userId
Chaque streamer a ses propres instances (`PlaybackQueue`, `TwitchBotManager`, etc.) dans `TenantManager`. Pas de partage d'état entre tenants.
